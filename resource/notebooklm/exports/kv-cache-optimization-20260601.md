## 核心主题

这个 notebook 聚焦大语言模型推理阶段的 KV Cache 压缩、调度、量化、卸载和架构级优化。主线是：长上下文和高并发会让 KV Cache 成为显存与带宽瓶颈，工程上需要在吞吐、延迟、精度和成本之间做取舍。

## 可拆分公众号选题

### 1. 为什么大模型越聊越卡？

- 核心问题：KV Cache 是什么，为什么上下文越长，显存占用越快增长。
- 普通读者切入：把对话记忆比作读长文时不断增加的笔记，笔记越多，桌面越挤。
- 技术点：自回归生成、重复计算、KV Cache、显存带宽瓶颈。
- 边界：不要把内存占用写成某个模型或厂商的工程错误，这是 Transformer 推理机制带来的通用压力。

### 2. 像管理电脑内存一样管理 AI 记忆

- 核心问题：为什么变长请求会导致显存碎片，PagedAttention 如何提高显存利用率。
- 普通读者切入：餐厅排座位，整桌预留会浪费，把大桌拆成小座位能提高翻台效率。
- 技术点：vLLM、PagedAttention、虚拟内存分页、Copy-on-Write、吞吐与延迟权衡。
- 边界：不要说 PagedAttention 在所有场景都最优；低并发极低延迟场景可能有额外查表开销。

### 3. 大模型记忆里的二八定律

- 核心问题：哪些 token 必须留下，哪些上下文可以丢弃。
- 普通读者切入：人读长文章也会抓主题、开头、最近内容和关键词。
- 技术点：Attention Sinks、Heavy Hitters、H2O、Scissorhands、重要性持久化。
- 边界：只讨论精度与工程取舍，不写成绕过系统提示或攻击提示词的教程。

### 4. 给 AI 的记忆做高压瘦身

- 核心问题：如何把 KV Cache 压到 4-bit、2-bit，同时尽量不损失质量。
- 普通读者切入：类似图片压缩，但 AI 记忆里存在离群值，不能简单一刀切。
- 技术点：KVQuant、KIVI、Key 的离群通道、Value 的平滑分布、RotateKV、Hadamard 旋转。
- 边界：不要说完全无损，应写“近乎无损”或“在特定任务上影响很小”。

### 5. 用 SSD 扩展大模型脑容量

- 核心问题：能否把 KV Cache 卸载到 SSD，从而处理更长上下文。
- 普通读者切入：显存像桌面，SSD 像仓库；仓库很大，但频繁搬运会慢。
- 技术点：HBM-DRAM-SSD 三级存储、NVMe SSD、GPU Direct Storage、Tutti、小随机 I/O。
- 边界：不要写成买大 SSD 就能跑任意大模型，I/O 和总线仍可能成为瓶颈。

### 6. DeepSeek MLA 与压缩的代价

- 核心问题：架构级 KV 压缩如何降低推理成本，激进压缩为什么会伤害复杂推理。
- 普通读者切入：便宜 API 背后不只是价格战，也有底层架构效率。
- 技术点：DeepSeek-V2 MLA、低秩投影、RoPE 解耦、CoT、GSM8K、MATH-500、ShotKV。
- 边界：避免拉踩任何模型。准确率下降应归因于极端压缩策略与任务类型，而不是某个模型“变笨”。

## 必须保留的事实点

- 未优化时，长上下文 KV Cache 会带来极高显存压力；具体数字必须限定模型规模、上下文长度和是否启用优化。
- 注意力矩阵在推理中常呈现高稀疏性，但这不等于模型参数稀疏。
- PagedAttention 主要解决多并发服务中的显存碎片与利用率问题，不等同于单请求速度一定提升。
- vAttention 相关材料强调连续虚拟地址在低并发延迟上的价值，适合写成系统设计权衡。
- Attention Sinks 保留前几个 token 能稳定流式生成，但不代表模型记住了所有被丢弃的中间内容。
- Key 和 Value 的分布不同，量化策略也应不同；Key 更容易受离群值影响。
- MLA 是架构级优化，不是能直接装到任意已有模型上的推理插件。
- 数学、逻辑推理和长链路 CoT 对激进缓存驱逐更敏感。
- 持久化 KV Cache 能在固定上下文高频复用场景降低 prefill 成本，但不能宣称 RAG 已经被淘汰。
